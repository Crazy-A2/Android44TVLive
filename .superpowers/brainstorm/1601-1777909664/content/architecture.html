架构方案对比

选择适合 Android 4.4.2 电视直播应用的整体架构

A

MVP + 单 Activity 多 Fragment

参考文档推荐的方案。一个主 Activity 管理所有 Fragment（频道列表、播放、设置等），Presenter 处理逻辑。Leanback 库原生支持此模式。

B

MVP + 多 Activity

每个功能模块独立 Activity（频道页、播放页、设置页）。简单直接，4.4.2 兼容性最好，但页面间传值稍复杂。

C

MVVM + 单 Activity

用 ViewModel + LiveData 管理状态。更现代但需要 Architecture Components 旧版（1.x），4.4.2 兼容性需额外验证。

权衡分析

方案A优势

  • Leanback 库原生适配
  • 页面切换流畅
  • 社区示例多

方案A劣势

  • Fragment 生命周期复杂
  • 单 Activity 内存压力大（4.4.2 内存有限）

方案B优势

  • 最简单可靠
  • 内存友好（可释放不需要的页面）
  • 4.4.2 零兼容问题

方案B劣势

  • 页面切换不够流畅
  • 传值稍繁琐

方案C优势

  • 状态管理更清晰
  • 数据驱动 UI

方案C劣势

  • 4.4.2 兼容风险高
  • 增加依赖复杂度
  • Leanback 适配需额外工作
